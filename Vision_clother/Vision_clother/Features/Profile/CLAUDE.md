# Features/Profile Module

User-profile: account sign-in/out, portrait capture, and entry points into taste-calibration tools and Insights.
- **Taste analytics moved to Insights:** `ProfileView` no longer computes or displays taste/color/formality/pairing/activity data itself — that's all in the Insights tab now (`Features/Insights/`). `ProfileView`'s `insightsLinkSection` just links there. Don't re-add narrative taste sections to `ProfileView` — extend Insights instead.
- **Sign-in/out lives in `AccountCardView`,** shown inline as the first section of `ProfileView`'s main list (not behind a sheet) — `AccountCardViewModel` owns the Google/phone sign-in flow and `signOut(syncCoordinator:)`.
- Account management that *isn't* sign-in/out — credit purchases, usage/quota readout, sync status, diagnostic log sharing, delete account — lives in `AccountSectionView`.
- **Settings Sheet:** `AccountSectionView` is hosted in a dedicated sheet (`isSettingsPresented`) triggered by the toolbar gear button — never in `ProfileView`'s main list flow. `AccountCardView`, by contrast, is deliberately inline (not sheeted) since it's the primary account-status surface now.
- **Modifier Placement In List:** Any presentation (`.sheet`, `.alert`) or lifecycle (`.task`) modifiers inside `AccountSectionView` (or `AccountCardView`) must be attached to a single leaf view (e.g., a zero-height `Color.clear` cell) rather than `Section`. Attaching sheet modifiers to `Section` causes SwiftUI to duplicate presentation attempts on every list item, corrupting the navigation/sheet context and causing sheets (like "Buy Credits") to automatically close.
- **Portrait Photo Flow:** User portrait captures are validated via `PersonPhotoValidationService` and saved to `UserPortraitStorage` + Firestore/Cloud Storage metadata (`users/{uid}/portrait/base_portrait.jpg`).
- **Default Body Photo:** "Use Default Image Instead" applies the bundled `Resources/DefaultBodyPhoto.jpg` (a mannequin photo) as the base try-on photo via `ProfileViewModel.useDefaultBodyPhoto()` — skips `PersonPhotoValidationService` (no human pose for Vision to detect) and `UserProfileDerivationService` (assigns a neutral `UserStyleProfileWire` instead), but is saved/uploaded through the exact same `UserPortraitStorage`/Cloud Storage path a real photo uses, so it syncs across devices for free. "Is the default active" is answered by byte equality (`UserPortraitStorage.isDefaultBodyPhoto`), not a separate persisted flag.
