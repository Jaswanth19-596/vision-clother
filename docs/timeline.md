## 2026-07-23 — Perf: downsampled ImageIO thumbnails for grid/list photo cells, separate from full-res detail cache

**Status:** Implemented. `xcodebuild -sdk iphonesimulator clean build` verified (BUILD SUCCEEDED); no test run per `CLAUDE.md` §2.

**Problem:** `Data/ImageStorage.swift`'s `cachedImage(for:)` always decoded the full-resolution on-disk garment/combination PNG via `UIImage(contentsOfFile:)` and cached the resulting bitmap in a 300-entry `NSCache`, regardless of how small the calling view actually rendered it. Every photo-bearing view — grid cells (`ClosetItemCell`, `PairingItemCell`), list rows (`CombinationRow`), and chips (`OutfitCardView`, rating/detail item swatches) — routes through the single wrapper `CachedWardrobeImage`, so at closet/history sizes in the thousands this was the dominant memory cost and a jetsam risk (`ClosetItemCell` renders an 84×84pt swatch but cached a full multi-megapixel decode for it).

**Fix:**
1. `Data/ImageStorage.swift`: new `cachedThumbnail(for:maxPixelSize:)`, decoding straight to a downsampled bitmap via `CGImageSourceCreateThumbnailAtIndex` (`kCGImageSourceCreateThumbnailFromImageAlways` + `kCGImageSourceThumbnailMaxPixelSize` + `kCGImageSourceCreateThumbnailWithTransform`) inside the same off-main `Task.detached` pattern `cachedImage(for:)` already uses — the full-res bitmap is never materialized. Cached separately from `imageCache` in a new `thumbnailCache: NSCache<NSString, ThumbnailBucket>`, one `ThumbnailBucket` (small locked `[pixelSizeKey: UIImage]` dict) per source filename so several on-screen sizes of the same photo share one cache slot and one invalidation call. `invalidateCache(_:)` and `wipeAll()` now clear both caches.
2. `DesignSystem/CachedWardrobeImage.swift`: new optional `thumbnailSize: CGSize?` param (default `nil` = unchanged full-res behavior). When set, multiplies by `@Environment(\.displayScale)` to get the target pixel size and calls `cachedThumbnail`; otherwise calls `cachedImage` as before.
3. Every grid/list call site now passes `thumbnailSize:` matching its actual frame: `ClosetView.swift` (`ClosetItemCell`, 84×84), `WardrobeInsightsView.swift` (`itemChip`, 56×56), `CombinationsView.swift` (`CombinationRow` thumbnail 60×60, inline item swatch 26×26), `CombinationDetailView.swift` (item chip, 44×44), `ManualPairingView.swift` (`PairingItemCell`, 100×100), `OutfitCardView.swift` (item swatch, 44×44), `RateCombinationView.swift` (item chip, 44×44). Left unset (full-res) at true detail/zoom hero sites: `ItemDetailView.swift`, `RateItemView.swift`, `CombinationDetailView.swift`'s `ZoomableImageContainer` render, `RateCombinationView.swift`'s zoomable render.

**File changes:** `Vision_clother/Data/ImageStorage.swift`, `Vision_clother/Vision_clother/DesignSystem/CachedWardrobeImage.swift`, `Vision_clother/Vision_clother/Features/Closet/ClosetView.swift`, `Vision_clother/Vision_clother/Features/Insights/WardrobeInsightsView.swift`, `Vision_clother/Vision_clother/Features/Combinations/CombinationsView.swift`, `Vision_clother/Vision_clother/Features/Combinations/CombinationDetailView.swift`, `Vision_clother/Vision_clother/Features/Pairing/ManualPairingView.swift`, `Vision_clother/Vision_clother/Features/DailyAssistant/OutfitCardView.swift`, `Vision_clother/Vision_clother/Features/Rating/RateCombinationView.swift`.

## 2026-07-23 — Refactor: centralized Cloud Functions scaling config, scaled down to early-dev defaults (~10 users)

**Status:** Implemented. Backend not build-verified (no `npm`/`tsc` run this session, per standing convention); not yet deployed.

**Problem:** The prior pass (previous entry below) hardcoded production-scale `cpu`/`memory`/`concurrency`/`maxInstances` directly in each `onRequest(...)` call in `index.ts`. Actual early-development traffic is ~10 active users total, not 1,000 — that capacity is unnecessary cost right now, and retuning it meant editing function definitions directly with no single place to see or adjust the numbers.

**Fix:** New `backend/functions/src/config/scaling.ts` exports `scalingConfig`, a `Record<"proxyApi"|"heavyApi"|"accountApi", Pick<HttpsOptions, "cpu"|"memory"|"concurrency"|"maxInstances"|"minInstances">>` — the single place capacity is defined. `index.ts`'s three `onRequest(...)` calls spread `...scalingConfig.<name>` instead of hardcoding these five fields; `timeoutSeconds`, `secrets`, and `invoker: "public"` stay in `index.ts` (unrelated to scaling). Values set to early-dev defaults: `proxyApi` (256MiB, concurrency 5, maxInstances 3), `heavyApi` (512MiB, concurrency 2, maxInstances 2), `accountApi` (256MiB, concurrency 5, maxInstances 3) — all `cpu: 1` (still required for `concurrency > 1` to take effect at all, per the previous entry's finding) and `minInstances: 0` (scale-to-zero, the right tradeoff at this traffic level). The prior production-scale numbers (512MiB–1GiB, concurrency 40–80, maxInstances 20–30, ~4,800 in-flight ceiling) are preserved only as a documented reference point in `docs/backend/architecture.md` for when traffic grows — not left live in code.

**File changes:** `backend/functions/src/config/scaling.ts` (new), `backend/functions/src/index.ts`, `docs/backend/architecture.md`, `backend/README.md`.

## 2026-07-23 — Fix: Cloud Functions concurrency/CPU bottleneck capping throughput at 60 in-flight requests

**Status:** Implemented. Backend not build-verified (no `npm`/`tsc` run this session, per standing convention); not yet deployed.

**Problem:** `backend/functions/src/index.ts`'s three v2 functions (`proxyApi`, `heavyApi`, `accountApi`) set no explicit `cpu` or `concurrency`. Cloud Functions v2 silently forces `concurrency` to 1 whenever CPU is fractional (the default at the 256MiB/512MiB memory tiers these were on), so each container instance could only serve one request at a time — real system-wide capacity was just `maxInstances` summed across the three functions (30 + 10 + 20 = 60 simultaneous in-flight requests), far short of the 1,000-concurrent-user target.

**Fix:** Each function now sets `cpu: 1` (unlocking real per-instance concurrency) and an explicit `concurrency` value, with memory bumped one tier since concurrent requests on one instance now share that instance's memory: `proxyApi` (256MiB→512MiB, concurrency 80, maxInstances unchanged at 30), `heavyApi` (512MiB→1GiB, concurrency 40 — lower than the other two since its work per-request is heaviest, maxInstances 10→20), `accountApi` (256MiB→512MiB, concurrency 80, maxInstances unchanged at 20). `timeoutSeconds`, `secrets`, and `invoker: "public"` unchanged on all three. New max in-flight capacity (`maxInstances × concurrency` summed): 30×80 + 20×40 + 20×80 = **4,800** (up from 60).

**File changes:** `backend/functions/src/index.ts`, `docs/backend/architecture.md` (Cloud Functions table), `backend/README.md` (per-function config parentheticals).

## 2026-07-23 — Fix: 4 production-blocking bugs in the credit engine, ahead of App Store submission

**Status:** Implemented. Backend not build-verified beyond `vitest` (no `npm run build`/`tsc`/emulator run this session, per standing convention); not yet deployed.

**Problem:** A pre-submission review of the just-built credit & tier management engine (previous entry below) surfaced four issues: (1) **Apple IAP compliance risk** — a PRO subscriber's monthly `autoReset` did `credits_remaining = tierConfig.creditAllocation`, a flat overwrite of the *same* field IAP top-ups incremented, so an unspent paid top-up was silently wiped at the next billing anniversary; (2) **rules/config desync** — `firestore.rules`' `itemCap()` hand-duplicated `pricing.config.ts`'s tier numbers on the stated (incorrect) premise that Security Rules can't read `config/pricing`; (3) a 5-minute pricing-config cache TTL meant ops changes took up to 5 minutes to reach every warm instance; (4) `refundCredit` used a plain non-transactional `.set()`.

**Fix:**
1. **Split wallet** (`middleware/creditGate.ts`, `routes/iapVerify.ts`, `routes/entitlementLimits.ts`, `types.ts`): `users/{uid}/meta/usage`'s `credits_remaining` field is replaced by two fields — `subscription_credits_remaining` (refilled to `tierConfig.creditAllocation` on `autoReset`, never additive) and `purchased_credits_remaining` (only ever touched by `iapVerify.ts`'s top-up grant or a debit/refund, never by billing reset). New exported `creditGate.ts` helper `totalCredits(subscription, purchased)`. Debits draw from `subscription_credits_remaining` first, `purchased_credits_remaining` for any remainder; `INSUFFICIENT_CREDITS` now compares against the combined total. The legacy pre-rewrite migration branch splits accordingly: derived headroom → subscription (floored at the tier allocation, as before), derived purchased balance → purchased (no longer folded into the floor, so it's never capped/lost). `req.quotaDebit` becomes `{ operation, subscriptionDebited, purchasedDebited }` so `refundCredit` can restore the exact bucket(s) debited; `refundCredit` now runs inside its own `runTransaction`. `iapVerify.ts` increments `purchased_credits_remaining` only. `entitlementLimits.ts` reports `creditsRemaining: totalCredits(...)` so the client's displayed balance is unaffected by the internal split.
2. **Rules/config desync** (`firestore.rules`): `itemCap()` now reads `config/pricing` live via `get(/databases/$(database)/documents/config/pricing)` — this is valid because Firestore Rules' `get()`/`exists()` calls are not subject to the *target* document's own `allow read`/`write` rules (that only gates client-initiated requests to that document directly; it doesn't block another rule's internal cross-document lookup). The previous hardcoded per-tier numbers are kept only as `itemCapFallback()`, used solely if `config/pricing` doesn't exist yet. A tier added only via `config/pricing` is now fully enforced at the rules layer immediately, no hand-edit/redeploy needed — this was previously documented as a permanent, accepted limitation; it wasn't actually permanent.
3. **Cache TTL** (`pricing.config.ts`): `CACHE_TTL_MS` 5 minutes → 1 minute, so an ops price/cap change reaches every warm instance in at most a minute. `modelAllowlist.ts`'s own TTL is unchanged (5 min) — deliberately different now, since it changes less often.
4. Tests updated: `test/creditGate.test.ts` (field renames throughout, new split-wallet-debit-order test, reset test now asserts `purchased_credits_remaining` survives untouched, both `refundCredit` tests updated to the two-bucket shape plus a new bucket-targeted-refund test), `test/pricing.config.test.ts` (TTL-boundary tests moved from `+5m1s` to `+1m1s`), `test/iapVerify.test.ts` (field renames, new test proving a top-up survives a subscription reset).
5. Docs: `docs/backend/architecture.md`, `docs/backend/conventions.md`, `backend/README.md`, and `backend/functions/src/iap/products.ts`'s doc comment all updated to the split-wallet field names, the 1-minute TTL, and the resolved (not permanent) rules/config desync.

## 2026-07-23 — Feature: dynamic, configurable credit & tier management engine

**Status:** Implemented. Backend not build-verified (no `npm`/`tsc` run this session, per standing convention — verified by re-reading every edited/new backend file for type correctness instead); not yet deployed.

**Problem:** The existing quota system (`entitlementLimits.ts`'s `TIER_LIMITS`/`ITEM_CAP_LIMITS`, `middleware/governance.ts`'s `governanceGate`) enforced fixed guest/free/premium monthly counters per feature (`recommendation`/`tryOn`), plus separate purchased-balance buckets per feature (`iap/products.ts`'s `PRODUCT_GRANTS`, `purchasedRecommendationBalance`/`purchasedTryOnBalance`) — changing a tier's limits or adding a new tier required editing TS constants and redeploying, and there was no single fungible "credit" concept a purchase or an operation cost could be expressed in generically. User asked for a genuine credit-accounting engine: one `credits_remaining` pool per user, per-operation costs (`UPLOAD`/`IMAGE_GEN`/`RECOMMENDATION`), and tier definitions (FREE one-time welcome pack, PRO recurring monthly allowance, plus any future tier) fully driven by a config an ops person can edit without touching backend logic or redeploying.

**Fix:**
1. New `backend/functions/src/pricing.config.ts` — replaces `entitlementLimits.ts`. Exports `OperationType` (`UPLOAD`/`IMAGE_GEN`/`RECOMMENDATION`), `TierConfig` (id, display name, monthly price, credit allocation, `autoReset`, optional per-operation hard caps, wardrobe item cap), `DEFAULT_OPERATION_COSTS`/`DEFAULT_TIER_CONFIGS` (seeded with `GUEST`/`FREE`/`PRO`), and `getPricingConfig`/`getOperationCost`/`getTierConfig`. Firestore-backed (`config/pricing`), following `modelAllowlist.ts`'s exact established pattern (`docs/backend/conventions.md`'s explicit template): Admin-SDK-only doc, hardcoded fallback for zero-seeding correctness, 5-minute warm-instance TTL cache, stale-cache-preferred-on-Firestore-failure (never fails open).
2. New `backend/functions/src/middleware/creditGate.ts` — replaces `governance.ts`'s `governanceGate`/`refundQuota`. `creditGate(operation)` runs, always transactionally (no fast-path cache — costs/tiers are now config-driven and can change at ops pace, so the old fixed-safety-margin cache trick doesn't generalize; an accepted latency/Firestore-cost tradeoff confirmed with the user): lazy init/migration for a doc with no `tier_id` yet → billing reset for recurring tiers past their `billing_cycle_start` monthly anniversary (rolling UTC-calendar-month add, not a fixed-day-count window — confirmed with the user) → hard-cap check → credit-balance check → debit + increment on success. `refundCredit` undoes exactly the recorded cost via `FieldValue.increment`, preserving `idempotency.ts`'s existing refund-on-failure contract. `middleware/governance.ts` is trimmed to just `rateLimitOnly` (the coarse daily request cap, unrelated to credits) plus the now-exported `usageRefFor`/`entitlementRefFor` doc-ref helpers shared with `creditGate.ts`.
3. Schema: `users/{uid}/meta/usage` gains `tier_id`, `credits_remaining`, `billing_cycle_start`, `usage_counts` (a map, one entry per operation type), `welcome_pack_claimed` — tier is folded in from the old separate `meta/entitlement.tier` field (now legacy, read-only, unwritten). Migration is lazy/per-user/exactly-once, inside `creditGate`'s own transaction: a legacy doc's leftover count-based headroom (`recommendationCount`/`tryOnCount` vs the old fixed limits) converts to credits at today's operation costs, any purchased balance (`purchasedRecommendationBalance`/`purchasedTryOnBalance`) is carried straight across 1:1 as now-fungible credits, floored at the resolved tier's own allocation so no legacy user is credited less than a brand-new signup would be; a genuinely fresh account (no usage doc at all) just gets the tier's allocation directly, no formula. `welcome_pack_claimed` is set `true` at that same moment either way, so nobody double-claims.
4. `types.ts`'s `AuthedRequest.quotaDebit` becomes `{ operation: OperationType; cost: number }` (was `{ feature: QuotaFeature; kind: "count"|"purchased" }` — one kind of debit now, so `kind` drops and `cost` is carried for an exact refund). `app.ts`: `/openrouter/recommend` → `creditGate("RECOMMENDATION")`; `/openrouter/tryon` and `/openrouter/images` → `creditGate("IMAGE_GEN")` (absorbing what the old `tryOn` feature covered — no fourth operation type introduced). `idempotency.ts`'s `idempotencyGate` is retyped from `QuotaFeature` to `OperationType` and now imports `refundCredit` instead of `refundQuota`. `routes/entitlementLimits.ts` response shape changes to `{tier, creditsRemaining, creditAllocation, operationCosts, operationCaps, billingCycleStart, autoReset, itemCap}` (was `{tier, recommendationLimit, tryOnLimit, itemCap}`); stays the same justified business-logic-exception route, just describing new numbers.
5. `iap/products.ts`'s `PRODUCT_GRANTS` drops `creditType`/`CreditType`, becoming flat `{ amount: number }` grants — purchased credits are now fully fungible, spendable on any operation type (confirmed with the user as an intentional behavior change from the old per-feature-scoped balances). `routes/iapVerify.ts` credits `credits_remaining` directly instead of a `creditType`-keyed balance field; response drops `creditType`.
6. `backend/firestore.rules`: new `config/pricing` block (Admin-SDK-only, mirrors `config/openrouterModels`). `itemCap()` rewritten from an `isPremium()`-ternary reading `meta/entitlement.tier` to a `resolveTierId()` helper reading `meta/usage.tier_id` (GUEST/FREE/PRO explicit branches, unrecognized `tier_id` — e.g. a tier added only via `config/pricing` — falls back to FREE's numbers as the safe floor). This is a **known, permanent, accepted limitation**: a dynamically-added tier gets full credit-engine enforcement immediately (Cloud Functions read `config/pricing` live), but its wardrobe item-cap enforcement at the rules layer needs a hand-edit + redeploy regardless, since Firestore Security Rules have no module system and can't read an arbitrary ops-editable Firestore doc for this purpose either.
7. Tests: new `test/pricing.config.test.ts` (mirrors `modelAllowlist.test.ts`'s `vi.resetModules()`/`freshModule()` pattern — default fallback, Firestore override, stale-cache-on-failure, TTL re-read, plus the "add a tier only in the mocked doc" proof) and `test/creditGate.test.ts` (unauthenticated, fresh FREE/GUEST init, legacy migration incl. the fresh-vs-legacy distinction caught while writing these tests — see below, billing reset only for `autoReset` tiers, hard cap, insufficient credits, exact debit/refund, fail-open/fail-closed, plus the two required "config change → behavior change, no code change" proofs: mutating `operationCosts.RECOMMENDATION` and adding a brand-new `ULTRA_PRO` tier, both in the mocked Firestore doc only). `test/governance.test.ts` trimmed to just `rateLimitOnly` (its `governanceGate`/`refundQuota` sections deleted, superseded by `creditGate.test.ts`). `test/iapVerify.test.ts` updated for the fungible-credit response/field shape.
8. Docs: `docs/backend/architecture.md` ("Governance middleware" section replaced with "Credit gate middleware", route table, `/entitlement/limits` row), `docs/backend/conventions.md` (new "Credit & tier management engine config" section citing `pricing.config.ts` as the second adopter of the `modelAllowlist.ts` config pattern), `backend/README.md` (route descriptions, Firestore section, testing section).

**A real bug caught during test-writing, not by inspection:** the initial migration formula didn't distinguish "usage doc exists but has zero legacy counts" (a real legacy user) from "no usage doc at all" (a genuinely fresh signup) — both hit the same "derive credits from legacy headroom" branch, which would have over-credited a brand-new signup (deriving credits from the *full* legacy tier limits as if they were unused headroom) well beyond their tier's actual welcome-pack allocation. Fixed by adding a third branch: a fresh account (no doc) grants exactly `tierConfig.creditAllocation`, no formula; the headroom-conversion formula only runs when a legacy doc genuinely exists.

**Scope decisions (confirmed with user before implementing):** full replace of the guest/free/premium system, not a coexisting v2 (`entitlementLimits.ts` deleted outright). Firestore-backed config with a code fallback (mirrors `modelAllowlist.ts`), not a plain static TS file — ops can retune costs/add tiers live. UPLOAD is defined in config/gatekeeper only this round — no new upload-authorization route; wardrobe uploads still go straight to Cloud Storage, unchanged. Purchased credits become fully fungible (a real, intentional behavior change for anyone who'd bought a feature-scoped pack under the old model). Billing reset is a rolling anniversary from `billing_cycle_start` (UTC calendar-month add), not a calendar-month-boundary reset like the old `periodKey`. Always-transactional gatekeeper, no warm-instance fast-path cache for credits (an accepted latency/Firestore-cost regression traded for correctness under dynamic config) — `rateLimitOnly`'s existing daily-cap cache is untouched.

**File changes:** `backend/functions/src/pricing.config.ts` (new), `backend/functions/src/middleware/creditGate.ts` (new), `backend/functions/src/middleware/governance.ts` (trimmed to `rateLimitOnly`), `backend/functions/src/entitlementLimits.ts` (deleted), `backend/functions/src/routes/entitlementLimits.ts`, `backend/functions/src/app.ts`, `backend/functions/src/middleware/idempotency.ts`, `backend/functions/src/types.ts`, `backend/functions/src/iap/products.ts`, `backend/functions/src/routes/iapVerify.ts`, `backend/firestore.rules`, `backend/functions/test/pricing.config.test.ts` (new), `backend/functions/test/creditGate.test.ts` (new), `backend/functions/test/governance.test.ts` (trimmed), `backend/functions/test/iapVerify.test.ts`, `docs/backend/architecture.md`, `docs/backend/conventions.md`, `backend/README.md`. iOS-side (`Services/EntitlementLimitsService.swift`, `Data/UsageTracker.swift`, `Data/Sync/FirestoreDTOs.swift`'s `UsageDTO`, `Domain/CreditPack.swift`) intentionally left unchanged — flagged as required follow-up, out of scope for this backend-only pass.

## 2026-07-21 — Security fix: exact-match OpenRouter model allowlist (Firestore-backed)

**Status:** Implemented. Backend not build-verified (no `npm`/`tsc` run this session, per standing convention — verified by re-reading every edited/new backend file for type correctness instead); not yet deployed.

**Problem:** Security review found `backend/functions/src/routes/openrouterChat.ts` and `openrouterImages.ts` accepted any non-empty `model` string (`z.string().min(1)`, `.passthrough()`, no allowlist) and proxied it to OpenRouter with the project's own API key. Combined with two other confirmed gaps — `AuthService.ensureGuestSession()` (`Vision_clother/Services/AuthService.swift:99-124`) lets a client mint unlimited free Firebase anonymous UIDs with no client-side throttle, and `middleware/governance.ts`'s rate limit/quota are keyed only by `uid` with no IP/device correlation — a scripted attacker could mint guest accounts indefinitely and drive real OpenRouter spend by always requesting the most expensive available model, each account's nominal cap trivially resettable. IP/device-based throttling for guests and enabling Firebase App Check were both scoped out of this pass (App Check remains blocked on a paid Apple Developer account, already tracked in `backend/README.md`).

**Fix:**
1. New `backend/functions/src/modelAllowlist.ts`: an **exact-match** allowlist (`assertModelAllowed(model, requestId)`), not a provider-prefix match — the goal is capping worst-case spend per request, and prefix-matching would still let an allowed provider's most expensive model through. Backed by Firestore (`config/openrouterModels`, `{ allowedModels: string[] }`) so ops can add a newly adopted model — `Vision_clother/Config/ModelConfig.swift`'s model names are Remote-Config-hotfixable with no app rebuild — without a backend redeploy. A hardcoded `DEFAULT_ALLOWED_MODELS` array (today's 4 active defaults + 3 documented alternates from `ModelConfig.swift`'s comments: `google/gemini-3.1-flash-lite`, `openai/gpt-5-mini`, `minimax/minimax-m3`, `google/gemini-3.1-flash-lite-image`, `qwen/qwen3-vl-30b-a3b-instruct`, `bytedance-seed/seedream-4.5`, `google/gemini-2.5-flash-image`) is the fallback-of-last-resort, so the feature works correctly with zero manual seeding — the doc is deliberately left unseeded for now. A module-scope, 5-minute-TTL in-memory cache (single global slot, same pattern as `governance.ts`'s warm-instance cache) avoids a Firestore read on every request; on a Firestore read failure, a stale-but-previously-valid cached value is preferred over the hardcoded defaults (so a transient outage doesn't un-allow a model ops added between deploys) — never fails open to "allow everything."
2. Both routes call `assertModelAllowed` right after their existing Zod body-shape check and before the upstream `fetch()`; a rejected model returns `403 { error: "model_not_allowed" }` (distinct from the existing `400 { error: "invalid_request_body" }`, since this is a well-formed, authenticated request rejected on policy, not a malformed one). No changes needed to `idempotency.ts`/`governance.ts` — the existing non-2xx-triggers-refund logic already refunds any quota debit for a `403` for free.
3. `backend/firestore.rules`: new top-level `match /config/openrouterModels { allow read, write: if false; }` block (Admin-SDK-only), alongside the existing `processedTransactions`/`idempotencyKeys` blocks — the first global (non-`users/{uid}`) Firestore config doc in this codebase.
4. Tests: `backend/functions/test/routes.test.ts` fixtures updated to allowlisted model strings (previously used non-existent placeholders like `"x"`/`"seedream"` that would now 403), plus new rejection-path tests for both routes; new `backend/functions/test/modelAllowlist.test.ts` covering default-fallback, Firestore-doc override, malformed-doc fallback, cache-hit/no-second-read, TTL expiry/re-read, and stale-cache-preferred-on-outage.
5. Docs: `docs/backend/architecture.md` (pipeline description + new "Model allowlist" subsection), `docs/backend/conventions.md` (new "Server-side model allowlist" subsection paired with the existing Remote Config hotfix section), `backend/README.md` (route descriptions + optional Firestore-seeding note).

**Scope decisions (confirmed with user):** exact-match allowlist, not provider-prefix — user's explicit direction, overriding an earlier provider-prefix design that had been proposed and reasoned through in plan mode before the user redirected. Firestore-backed with a hardcoded fallback, left unseeded. Backend-only — the iOS client is unchanged; a future pass may migrate model config ownership to the backend (client fetching from a backend endpoint instead of reading Remote Config directly), out of scope now.

**File changes:** `backend/functions/src/modelAllowlist.ts` (new), `backend/functions/src/routes/openrouterChat.ts`, `backend/functions/src/routes/openrouterImages.ts`, `backend/firestore.rules`, `backend/functions/test/routes.test.ts`, `backend/functions/test/modelAllowlist.test.ts` (new), `docs/backend/architecture.md`, `docs/backend/conventions.md`, `backend/README.md`.

## 2026-07-21 — Security fix: quota bypass via forgeable `responseCache` Firestore doc

**Status:** Implemented. Firestore rules + docs only, no build required (rules/doc-only change).

**Problem:** Security audit (`security_issues.md` Finding 1) found that `backend/functions/src/middleware/responseCache.ts` caches successful `/openrouter/recommend` responses at `users/{uid}/responseCache/{sha256(body)}` and is mounted **before** `governanceGate` in `app.ts`, so a cache hit skips the quota check and the paid OpenRouter call entirely. `backend/firestore.rules`'s catch-all rule granted the doc owner full read/write on any subcollection except `meta`/`wardrobeItems`, and `responseCache` fell through to owner-writable. Since the cache key is just `sha256(JSON.stringify(req.body))` of the client's own request, an attacker could precompute the key for an intended request, write a fabricated `{status: 200, body, expiresAt: <future>}` doc directly via the Firestore SDK, then call the real endpoint and have the forged response served with zero quota debit and zero upstream call — fully defeating the quota/purchased-balance system on that account (no cross-user leakage).

**Fix:**
1. `backend/firestore.rules`: added an explicit `match /responseCache/{document=**} { allow read: if isOwner(); allow write: if false; }` block, mirroring the existing `meta/usage`/`meta/entitlement` owner-read/server-only-write pattern. Updated the trailing catch-all rule to also exclude `"responseCache"` from its owner-write grant, and extended the file's header comment accordingly. No change to `responseCache.ts`/`app.ts` — its writes go through the Admin SDK, which bypasses security rules by design, so only client-side forgery is closed.
2. Docs: `docs/backend/architecture.md`'s "Not yet built" section now notes `responseCache`'s server-only-write treatment alongside the governance/entitlement/idempotency collections. `security_issues.md` Finding 1 marked fixed and its Pre-Production Launch Checklist item checked off.

## 2026-07-21 — Perf/reliability fix: batched Firestore outbox writes (`WriteBatch`) + bootstrap push no longer re-uploads clean rows

**Status:** Implemented. `xcodebuild clean build` — BUILD SUCCEEDED. No test run (project convention — no existing tests touched this sync layer either).

**Problem:** User asked for an analysis of why `WardrobeSyncCoordinator.pushEverythingLocal` fires on sign-in and how outbox pushes are structured, then asked for the resulting refactor. Two issues found: (1) `Data/SyncOutboxWorker.swift`'s `drainNow` pushed one row at a time via 11 near-identical `pushX(dto:uid:)` methods on `Services/WardrobeSyncService.swift` (5 concurrent individual `setData` calls, not batched) — extra round trips for a large backlog (bootstrap push, or catching up after being offline). (2) `WardrobeSyncCoordinator.markDirtyForBootstrap` unconditionally re-marked *every* local row dirty on every bootstrap attempt regardless of existing `SyncMetadata` state, so a partially-failed bootstrap (`pushEverythingLocal` returning `fullySynced == false`) caused every retry (`retrySync`/`reconcileIfSignedIn`'s self-heal) to reprocess and re-push the entire local dataset, not just the rows that actually failed — and re-stamped `localUpdatedAt` on already-clean rows in the process.

**Fix:**
1. `Vision_clother/Services/WardrobeSyncService.swift`: replaced the 11 `pushX(dto:uid:)` methods plus `deleteEntity` (all had exactly one caller) with a single `commitBatch(_ operations: [SyncBatchOperation], uid:)` on the `WardrobeSyncService` protocol. New `SyncBatchOperation` value type (`entityType`/`entityID`/`operation`/`payload`) mirrors `SyncMetadata`'s fields. `FirestoreWardrobeSyncService.commitBatch` resolves each operation's `DocumentReference`, decodes/Firestore-encodes its payload (or builds the tombstone dict for a `.delete`), and adds it to one `Firestore.WriteBatch` per call — one network round trip per chunk instead of per row. `maxBatchSize = 500` (Firestore's hard per-batch cap). `AuthGatedWardrobeSyncService`/`MockWardrobeSyncService` updated to the new single-method surface.
2. `Vision_clother/Data/SyncOutboxWorker.swift`: `drainNow` now chunks eligible dirty rows into groups of up to 500 and commits each chunk via `commitBatch`, with up to 4 chunks concurrent (`maxConcurrentBatches`, replacing the old `maxConcurrentPushes = 5` single-row fan-out). A `WriteBatch` commit is atomic per chunk: on success every row in the chunk is marked clean together (`finalizeSuccess`); on failure every row in the chunk gets its own `attemptCount`/`lastAttemptAt` bumped individually, preserving per-row exponential backoff even though the network call is now batched. Legacy `.swipeEvent` rows are drained locally before chunking (never occupy a batch slot).
3. `Vision_clother/Data/WardrobeSyncCoordinator.swift`: `markDirtyForBootstrap` now skips a row outright when its existing `SyncMetadata.isDirty == false`, leaving `localUpdatedAt` untouched — safe because a clean row surviving into a bootstrap pass can only be there from an earlier bootstrap attempt for this same uid (`wipeLocalMirror` already clears `SyncMetadata` entirely when switching from a *different* previously-mirrored account).
4. Docs: `docs/decisions/resolved-v1.md` (new "Batched Outbox Writes + Bootstrap Push Guard" section, refining the "Cloud Sync" entry's outbox bullet) and `Vision_clother/Data/CLAUDE.md` (Cloud Sync section) updated to describe the batching + bootstrap-guard behavior.

**File changes:** `Vision_clother/Services/WardrobeSyncService.swift`, `Vision_clother/Data/SyncOutboxWorker.swift`, `Vision_clother/Data/WardrobeSyncCoordinator.swift`, `docs/decisions/resolved-v1.md`, `Vision_clother/Data/CLAUDE.md`.

## 2026-07-18 — Fix: Firestore transaction contention in `rateLimit`/`quotaGate` middleware (CRIT)

**Status:** Implemented. Backend TypeScript not compiled/run in this session (standing instruction: no `npm`/`tsc`/emulator in `backend/functions` — verify by reading); tests updated but not executed per the same rule.

**Problem:** A code review flagged that every proxied request (`/openrouter/recommend`, `/openrouter/tryon`, `/openrouter/images`) ran two serialized Firestore *transactions* before the upstream call — `rateLimit.ts`'s manual read-modify-write on `rateLimits/{uid}_{date}`, and `quota.ts`'s read of `users/{uid}/meta/usage` + `users/{uid}/meta/entitlement` (including the purchased-credit drawdown). Both middlewares share a fail-open posture for linked accounts on any thrown Firestore error; under transaction contention Firestore throws ABORTED, which tripped that fail-open path for ordinary traffic — silently disabling the abuse guardrail for exactly the accounts that matter most. Since both hot docs are keyed per uid, the only real source of same-document contention is a single account's own overlapping requests (rapid double-taps, two signed-in devices), not cross-user load — which ruled out sharded counters or an external store (Redis/Upstash) as overkill; confirmed with the user to stay Firestore-native rather than add new infra.

**Fix:** `rateLimit.ts` drops `runTransaction` entirely — a plain non-transactional `get()` for the limit check, then a non-transactional atomic `FieldValue.increment(1)` write. `FieldValue.increment` doesn't throw/retry under concurrent writes the way a transaction does; the read+write pair is no longer atomic, so a same-uid burst can overshoot the 500/day cap by a small bounded amount, an accepted trade since the file already documents this as "not a billing/metering system," just a coarse guardrail. `quota.ts` splits into a fast path and the original slow path: a non-transactional pre-read decides which to take via `QUOTA_SAFETY_MARGIN = 3` (comfortably under the tier limit → fast path, a single `FieldValue.increment` write on just the count field) vs. the existing transaction (within the margin, over the limit, or a monthly rollover is due → slow path, byte-for-byte unchanged including the purchased-balance drawdown). The drawdown's no-double-spend guarantee against `routes/iapVerify.ts`'s concurrent grant transaction is untouched — it still only runs inside `runTransaction`, and the fast path never touches the balance field at all, so it can't race the grant. Both middlewares' fail-open/fail-closed catch blocks are structurally unchanged; what changes is that the common-case paths no longer have a transaction to abort, so the catch block reverts to catching genuine Firestore outages instead of routine same-uid contention.

**File changes:** `backend/functions/src/middleware/rateLimit.ts` (drop transaction, non-transactional increment), `backend/functions/src/middleware/quota.ts` (fast/slow path split, `QUOTA_SAFETY_MARGIN` constant), `backend/functions/test/rateLimit.test.ts` (mock rewritten for plain `get`/`set` + `FieldValue.increment` sentinel, new "never uses a transaction" case), `backend/functions/test/quota.test.ts` (mock rewritten so transactional and non-transactional ref calls share the same fake state; new fast-path/slow-path/rollover boundary cases). ## 2026-07-14 — Phase 1: restructured the StylistBrain recommendation prompt + fixed the interview/bag formality bug

**Status:** Fixed (Phase 1 of 2 — see `docs/decisions/stylist-intelligence-engine.md`'s 2026-07-14 addendum for the full writeup; Phase 2, multi-accessory support, is planned separately, not implemented).

Two problems: (1) an "interview" recommendation surfaced a bag — a formality mismatch nobody would actually wear — and (2) an independent review of the ~230-line `Domain/StylistBrain.swift` prompt found it mixed policy with data, read as a step-by-step chain-of-thought script, and gave the `confidence` field (already in the wire schema, never explained) no real guidance.

Root cause of (1) turned out to be **two separate biases stacking**: the prompt's only accent-slot heuristic was a coarse "errands/travel/work suggests bag," with nothing re-checking that the specific bag chosen actually cleared the scenario's formality bar; and — caught mid-implementation via user's own diagnosis — `outfitRecommendationJSONSchema` uses `strict: true`, which forces every property (including the four semantically-optional accent/layer slots) into the JSON Schema's `required` array, since strict mode has no concept of "required key, optional value."
