# Stylist Intelligence Engine (2026-07-10)

Extends, does not supersede, the "LLM-as-Recommender Reversal" in `resolved-v1.md` — the LLM remains the primary recommender over a bounded text catalog, with `Domain/OutfitRecommendationValidator.swift` and `Domain/OutfitRecommendationEngine.swift` as the deterministic trust boundary and permanent offline fallback.

## Why

The recommendation pipeline works, but the knowledge driving it lived entirely as unversioned prose strings in `Domain/StylistBrain.swift`, duplicated conceptually against the numeric rules in `Domain/PairCompatibilityScoring.swift` with nothing keeping the two in sync. Separately, the LLM path passed `constraints: nil` to the validator, so dress-code (formality) alignment was only ever prompt guidance, never a deterministic check — and the learned taste profile injected into the prompt was positives-only, discarding every disliked attribute the feedback system had already computed.

## Decisions (Phase 1 — implemented)

- **Decision Hierarchy is lexicographic, not blended.** `StylistBrain.DecisionHierarchy` tiers now declare an `enforcement` (`.reject`, `.penalize`, or `.guide`). A lower tier always dominates a higher one — a beautiful outfit that breaks the dress code is wrong, not low-scoring. Only the lowest tier (aesthetic trend) is `.guide`: prompt-only, tie-breaking, never able to override a rejected/penalized tier.
- **Shared constants table, not duplicated literals.** `Domain/FashionKnowledgeConstants.swift` holds the formality-mismatch thresholds that both `StylistBrain`'s prompt prose and `PairCompatibilityScoring.aestheticPrior`'s scoring now read from — one number, two projections, verified by `StylistBrainTests`'s cross-projection consistency test.
- **Resolved constraints are self-reported, not re-extracted.** `OutfitRecommendationResponse` gained an optional top-level `resolvedConstraints: StyleConstraints` field. The recommendation prompt now asks the model to report what it resolved the scenario's dress code/weather/season to be *in the same call* that picks items — not a second call to `IntentExtractionService`, which must stay off the AI happy path (an existing tested invariant: `DailyAssistantViewModelTests.happyPathUsesRecommendationServiceAndSkipsTheFallbackEngine` asserts `intentService.callCount == 0`). `DailyAssistantViewModel` threads `response.resolvedConstraints` into the validator, so Tier 1 formality alignment is finally enforced on the LLM path.
- **Taste injection is symmetric.** `StylistBrain`'s prompt previously surfaced only attributes with affinity > 0.6 ("prioritize these"). It now also surfaces affinity < 0.4 ("tends to dislike") — the strongest personalization signal the feedback system computes, previously discarded before it ever reached the model.
- **Rejections carry a reason.** `OutfitRecommendationValidator.validateVerbose(...)` returns a `RejectionReason` (`.unknownID`, `.wrongSlot`, `.duplicateID`, `.ghostElement`) per dropped pick, additive alongside the unchanged `validate(...)`. Not wired to telemetry yet — it exists so a future "why did the LLM path yield nothing" investigation has data instead of a silent `compactMap`.
- **Fashion Constitution honesty clause.** The prompt now explicitly tells the model to say so when the wardrobe genuinely lacks a suitable item, rather than forcing a poor match, and to use respectful, non-judgmental body/appearance language.

## Decisions (Phase 1 addendum — dimension-based outfit feedback, 2026-07-10)

Replaces the outfit-level "Rate this outfit" flow's blended Fit/Comfort/Confidence/Wear-again form with dimension-based questions, each mapped to a specific part of the recommendation engine rather than one generic satisfaction score:

- **`OutfitFeedback` is a clean-break redesign, not additive.** No shipped users existed on the old shape, so `Models/FeedbackEvent.swift` replaces the outfit-level directional `fit`/`comfort`/`confidence`/`wearAgain: Bool` fields outright with Level 1 (Overall Satisfaction, Wear Again as `WearAgainAnswer` yes/maybe/no, Confidence, Comfort) and Level 2 (Occasion Match, Personal Style Match, Color Harmony, Fit & Silhouette, Weather Suitability, Practicality) plus a Favorite/Weakest Item pick — all still optional, `nil` for the simple auto-recorded "liked" write path. Item-level `ItemRating`/`FitRating` are unchanged (a future phase).
- **Every Level 2 dimension is wired to a specific affinity, not captured-and-deferred.** `Domain/AttributePreferenceProfile.swift` gained three new Bayesian-shrunk affinity maps — `styleTagAffinity` (Personal Style Match, keyed by `WardrobeItem.styleTags`), `silhouetteAffinity` (Fit & Silhouette, keyed by `WardrobeItem.silhouette`), `fabricWeightAffinity` (Weather Suitability averaged with Practicality — both describe "does this garment work for the conditions," folded into one bucket rather than inventing an 8th dimension with no scoring hook). Color Harmony and Occasion Match fold into the existing `colorVibeAffinity`/`formalityAffinity` maps alongside `ItemRating`'s contribution. All six affinities feed `affinityBonus(for:)` (now averaged over 6 dimensions, was 3) and `StylistBrain`'s symmetric taste-injection prompt block, identically to the existing color/pattern/formality treatment.
- **Favorite/Weakest Item reuses the existing item-preference channel.** No new scoring math — `WardrobeRepository.fetchFeedbackHistory()` folds a favorite pick as an extra like and a weakest pick as an extra dislike into the same `itemFeedback` counts `PairCompatibilityScoring.itemPreference` already reads.
- **`SavedCombination` now persists all 4 slots.** Previously only `topItemID`/`bottomItemID` were durable, even though Daily Assistant's in-memory `OutfitCombination` always resolves footwear (and optionally outerwear). Added `footwearItemID`/`outerwearItemID` (+ denormalized labels), populated by `DailyAssistantViewModel.saveCombination()`; `nil` for Manual Pairing saves, which only ever select a top+bottom. This is what lets the Favorite/Weakest Item picker offer every real slot in the outfit.
- **Not wired this phase (deferred):** the "What would you change?" checklist (Level 3), the item-level rating overhaul (Versatility/Frequency/Style Identity/Quality Perception/etc.), and the explicit accept/edit/reject preference-confirmation card — see Deferred below.

## Deferred (later phases)

Full knowledge-module extraction (separate Color Theory / Fabric Science / Dress Code / Silhouette / Footwear modules with individual applicability predicates and prompt-budget eviction), the validator's contextual (Tier B) and repair (Tier C) tiers, a named versioned Recommendation Rubric with per-dimension subscores, exemplar-based taste injection, impression/selection event capture, and controlled vocabularies for the rich garment attributes are all real next steps but were not done in this pass — each touches either the live structured-output contract with `google/gemini-3.1-flash-lite` or the well-tested numeric scoring engine, and is better done incrementally with its own regression coverage rather than in one sweep. See the full architecture design for the target end-state; this file tracks only what's actually shipped.
