# Stylist Intelligence Engine (2026-07-10)

Extends, does not supersede, the "LLM-as-Recommender Reversal" in `resolved-v1.md` — the LLM remains the primary recommender over a bounded text catalog, with `Domain/OutfitRecommendationValidator.swift` and `Domain/OutfitRecommendationEngine.swift` as the deterministic trust boundary and permanent offline fallback.

## Why

The recommendation pipeline works, but the knowledge driving it lived entirely as unversioned prose strings in `Domain/StylistBrain.swift`, duplicated conceptually against the numeric rules in `Domain/PairCompatibilityScoring.swift` with nothing keeping the two in sync. Separately, the LLM path passed `constraints: nil` to the validator, so dress-code (formality) alignment was only ever prompt guidance, never a deterministic check — and the learned taste profile injected into the prompt was positives-only, discarding every disliked attribute the feedback system had already computed.

## Decisions (Phase 1 — implemented)

- **Decision Hierarchy is lexicographic, not blended.** `StylistBrain.DecisionHierarchy` tiers now declare an `enforcement` (`.reject`, `.penalize`, or `.guide`). A lower tier always dominates a higher one — a beautiful outfit that breaks the dress code is wrong, not low-scoring. Only the lowest tier (aesthetic trend) is `.guide`: prompt-only, tie-breaking, never able to override a rejected/penalized tier.
- **Shared constants table, not duplicated literals.** `Domain/FashionKnowledgeConstants.swift` holds the formality-mismatch thresholds that both `StylistBrain`'s prompt prose and `PairCompatibilityScoring.aestheticPrior`'s scoring now read from — one number, two projections, verified by `StylistBrainTests`'s cross-projection consistency test.
- **Resolved constraints are self-reported, not re-extracted.** `OutfitRecommendationResponse` gained an optional top-level `resolvedConstraints: StyleConstraints` field. The recommendation prompt now asks the model to report what it resolved the scenario's dress code/weather/season to be *in the same call* that picks items — not a second call to `IntentExtractionService`, which must stay off the AI happy path (an existing tested invariant: `DailyAssistantViewModelTests.happyPathUsesRecommendationServiceAndSkipsTheFallbackEngine` asserts `intentService.callCount == 0`). `DailyAssistantViewModel` threads `response.resolvedConstraints` into the validator, so Tier 1 formality alignment is finally enforced on the LLM path.
- **Taste injection is symmetric.** `StylistBrain`'s prompt previously surfaced only attributes with affinity > 0.6 ("prioritize these"). It now also surfaces affinity < 0.4 ("tends to dislike") — the strongest personalization signal the feedback system computes, previously discarded before it ever reached the model.
- **Rejections carry a reason.** `OutfitRecommendationValidator.validateVerbose(...)` returns a `RejectionReason` (`.unknownID`, `.wrongSlot`, `.duplicateID`, `.ghostElement`) per dropped pick, additive alongside the unchanged `validate(...)`. Not wired to telemetry yet — it exists so a future "why did the LLM path yield nothing" investigation has data instead of a silent `compactMap`.
- **Fashion Constitution honesty clause.** The prompt now explicitly tells the model to say so when the wardrobe genuinely lacks a suitable item, rather than forcing a poor match, and to use respectful, non-judgmental body/appearance language.

## Decisions (Phase 1 addendum — dimension-based outfit feedback, 2026-07-10)

Replaces the outfit-level "Rate this outfit" flow's blended Fit/Comfort/Confidence/Wear-again form with dimension-based questions, each mapped to a specific part of the recommendation engine rather than one generic satisfaction score:

- **`OutfitFeedback` is a clean-break redesign, not additive.** No shipped users existed on the old shape, so `Models/FeedbackEvent.swift` replaces the outfit-level directional `fit`/`comfort`/`confidence`/`wearAgain: Bool` fields outright with Level 1 (Overall Satisfaction, Wear Again as `WearAgainAnswer` yes/maybe/no, Confidence, Comfort) and Level 2 (Occasion Match, Personal Style Match, Color Harmony, Fit & Silhouette, Weather Suitability, Practicality) plus a Favorite/Weakest Item pick — all still optional, `nil` for the simple auto-recorded "liked" write path. Item-level `ItemRating`/`FitRating` are unchanged (a future phase).
- **Every Level 2 dimension is wired to a specific affinity, not captured-and-deferred.** `Domain/AttributePreferenceProfile.swift` gained three new Bayesian-shrunk affinity maps — `styleTagAffinity` (Personal Style Match, keyed by `WardrobeItem.styleTags`), `silhouetteAffinity` (Fit & Silhouette, keyed by `WardrobeItem.silhouette`), `fabricWeightAffinity` (Weather Suitability averaged with Practicality — both describe "does this garment work for the conditions," folded into one bucket rather than inventing an 8th dimension with no scoring hook). Color Harmony and Occasion Match fold into the existing `colorVibeAffinity`/`formalityAffinity` maps alongside `ItemRating`'s contribution. All six affinities feed `affinityBonus(for:)` (now averaged over 6 dimensions, was 3) and `StylistBrain`'s symmetric taste-injection prompt block, identically to the existing color/pattern/formality treatment.
- **Favorite/Weakest Item reuses the existing item-preference channel.** No new scoring math — `WardrobeRepository.fetchFeedbackHistory()` folds a favorite pick as an extra like and a weakest pick as an extra dislike into the same `itemFeedback` counts `PairCompatibilityScoring.itemPreference` already reads.
- **`SavedCombination` now persists all 4 slots.** Previously only `topItemID`/`bottomItemID` were durable, even though Daily Assistant's in-memory `OutfitCombination` always resolves footwear (and optionally outerwear). Added `footwearItemID`/`outerwearItemID` (+ denormalized labels), populated by `DailyAssistantViewModel.saveCombination()`; `nil` for Manual Pairing saves, which only ever select a top+bottom. This is what lets the Favorite/Weakest Item picker offer every real slot in the outfit.
- **Not wired this phase (deferred):** the "What would you change?" checklist (Level 3), the item-level rating overhaul (Versatility/Frequency/Style Identity/Quality Perception/etc.), and the explicit accept/edit/reject preference-confirmation card — see Deferred below.

## Decisions (Phase 1 addendum — prompt architecture restructure + accent-slot formality fix, 2026-07-14)

Two problems drove this pass: an "interview" recommendation surfaced a bag (accent items were never held to the Dress Code tier's formality bar), and independent review found the ~230-line prompt mixed policy with data and read as a step-by-step chain-of-thought script rather than a set of outcome requirements.

- **Decision Hierarchy compressed from 7 tiers to 6.** Color Harmony and Fit/Silhouette merge into one `.visualCohesion` tier — both were already `.penalize`-enforced with no behavioral distinction, so merging lets the prompt trade color vs. proportion against each other within one tier instead of rigidly ordering one over the other. Each tier's prose is now compressed into a Purpose/Priority/Never form rather than a paragraph.
- **Dress Code (Tier 2) now explicitly gates accent slots.** Previously the only accent-slot guidance was a coarse "errands/travel/work suggests bag" heuristic with no formality check on the specific item chosen — an interview scenario would loosely match "work," flag `bag` as desired, and pick any bag regardless of formality. The tier's prose now states the formality-mismatch check applies to every populated slot including accents, gives scenario-specific accent guidance (business/interview/formal → at most one subtle accessory, never headwear, bag only if a structured/formal option exists; outdoor/casual → headwear; errands/commute/travel → bag), and instructs the model to omit an accent slot rather than force a mismatch.
- **Explicit step-by-step "Reasoning Workflow" removed.** Replaced with one declarative sentence ("confirm every higher-priority tier is satisfied before considering a lower one") — modern models don't need an explicit chain-of-thought script, and the individual facts that script conveyed (resolved-constraints reporting, rationale length, confidence) moved to a plain OUTPUT FORMAT section.
- **Preferences (Tier 4) split into intrinsic profile vs. learned behavior**, and ratings are now explicitly scoped: "ratings only break ties among otherwise comparable candidates — never let a high rating substitute for correct formality, color, or fit."
- **`rationale.confidence` given real meaning.** The field already existed in the wire schema (`StructuredRationaleWire.confidence: Int`) and was fully plumbed through decode/validate, but was never explained to the model or bounded in the JSON schema. The prompt now specifies a calibrated 0-100 scale (lower when a tier required a compromise or the scenario was ambiguous), and `Services/OutfitRecommendationService.swift`'s schema gained `"minimum": 0, "maximum": 100` on that property.
- **Schema-level fix for the accent-slot bias, not just prompt prose.** `outfitRecommendationJSONSchema` uses `strict: true`, which forces every property — including the four semantically-optional accent/layer slots — into `required`; strict JSON Schema has no concept of "required key, optional value." That structural fact biases smaller models toward filling every accent slot with a plausible item just because its key is present (this is what put a formality-matching bag into an interview outfit even with the prompt's Tier 2 guidance in place). `itemIDSchemaProperties` (now `schemaProperty(for:)`) gives each optional slot's JSON Schema entry an explicit `description` separating "the key must exist" from "the value should usually be null," with slot-specific guidance (`bag_id`'s calls out interview/formal-business by name) — this is a second, schema-level backstop alongside the prompt-prose fix, since the prompt alone wasn't sufficient against strict mode's required-key bias.
- **New diversity and ranking objectives.** Outfits must differ by at least one primary garment or a clearly different styling direction (no near-duplicate color swaps), and `outfits` must be sorted strongest-to-weakest so index 0 is always the best recommendation.
- **Validator/architecture jargon removed from the prompt.** The `resolved_constraints` instruction no longer mentions "the on-device validator" — the model doesn't benefit from knowing the software architecture, only what it must populate and why it must be truthful.
- Not touched this pass: `OutfitRecommendationEngine`, `OutfitRecommendationValidator`, `Domain/FashionKnowledgeConstants.swift`, and multi-accessory support (letting an outfit hold several simultaneous accessories) — the latter requires a schema/engine/validator/SwiftData migration across ~12 files and is scoped as a separate future phase.

## Decisions (Item Rating question redesign + Swipe-to-Learn implicit swipe bridge, 2026-07-15)

User feedback on the per-item rating form (`Features/Rating/RateItemView.swift`): the questions read as generic and out of scope. Tracing the scoring path confirmed the deeper cause — every answer (Fit, Comfort, Confidence, Wear-again, Versatility, Predicted Wear Frequency, Style Identity, Quality Perception) was averaged into one blended `ItemRating.normalizedValue`, and that single number was reused as the signal for three unrelated affinities (color, pattern, formality) with no dedicated color or pattern question ever asked. Versatility/Frequency/Quality Perception fed nothing else — they only diluted the blend.

- **Question set redesigned to map 1:1 onto scoring dimensions.** `Models/ItemRating.swift` replaces Confidence/Versatility/Frequency/Quality Perception with `colorLike` ("do you like this color on you?"), `patternLike` (optional — the Pattern question is skipped for solid-pattern items), and `formalityFit` ("did this feel right for how formal/casual you needed it?"). Fit, Comfort, Style Identity, and Wear Again are unchanged. Clean-break schema change (no shipped users), matching the precedent set by the `OutfitFeedback` redesign above.
- **`RatedAttributes` (`Domain/AttributePreferenceProfile.swift`) carries dedicated per-attribute fields instead of one blended `value`.** `colorLike` feeds `colorVibeAffinity`, `patternLike` feeds `patternAffinity` (skipped when `nil`), `formalityFit` feeds `formalityAffinity`, `styleIdentity` still feeds `styleTagAffinity` (unchanged). Two attributes that previously had no item-level signal at all now do: `fit.centeredness` feeds `silhouetteAffinity` when the item has a `silhouette` tag, and `comfort` feeds `fabricWeightAffinity` — both alongside the existing outfit-level `Fit & Silhouette`/`Weather Suitability`+`Practicality` contributions.
- **Item ratings now close the loop into Swipe-to-Learn Visual Taste.** A highly-rated item (`ItemRating.impliesLiked`, the existing `>=0.6` threshold) is treated as an implicit "swipe right" on its cached photo embedding; a poorly-rated one as an implicit "swipe left" — nudging the same `VisualPreferenceState` centroids `Features/SwipeDiscovery/` writes (`Data/WardrobeRepository.swift`'s new `applyImplicitSwipe`, called from `recordItemRating`). This keeps the k-means visual profile learning from real wear history even after a user stops swiping the discovery deck, and keeps the text-attribute profile and the visual embedding space in sync from the same feedback moment.
- **Implicit swipes use a gentler, fixed learning rate than explicit swipes.** `VisualClusterUpdater.update` gained an optional `learningRate: Float?` parameter (`nil` preserves the existing `1/weight` incremental-mean step used by every pre-existing call site). `VisualClusterUpdater.implicitLearningRate = 0.05` is used only for rating-derived updates — a rating is a more passive, ambient signal than a deliberate swipe gesture and arrives far more often, so it should move the centroids less per event.
- **Implicit swipes are best-effort and don't write a `SwipeEvent`.** `applyImplicitSwipe` no-ops (silently, via `try?` at the `recordItemRating` call site) if the item has no cached `WardrobeItemEmbedding` yet — embeddings are computed lazily by `fetchFeedbackHistory()`, so a rating recorded before that happens simply misses one nudge rather than blocking the rating save on an async Vision computation. It also doesn't insert a `SwipeEvent`: a rating isn't a discrete stock-photo swipe, so `VisualPreferenceProfile.build(from:dislikedEmbeddings:)`'s history-replay recovery path stays scoped to actual swipe gestures.
- **`dislikePenaltyWeight = 1.2` and `maxBonusMagnitude = 0.3` in `Domain/VisualPreferenceProfile.swift` are unchanged** — this pass only adds a new writer into the existing centroid/scoring pipeline, not a new formula.

## Decisions (Calibration meter + `[AI-Stylist-ML]` verification logging, 2026-07-15)

A follow-up request asked for the swipe-deck's Vision-embedding/k-means pipeline to be built essentially from scratch, with a specific formula/parameter spec. Investigation found the entire pipeline already shipped (the two decisions above) — Pexels query pool, `VisualClusterUpdater`'s k-means, the catalog-truncation and re-scoring integrations all already matched the spec's numbers almost exactly (`dislikePenaltyWeight = 1.2`, `maxBonusMagnitude = 0.3`, `implicitLearningRate = 0.05`). Two spec details did *not* match already-shipped, documented behavior — nearest-centroid assignment via Euclidean distance (existing: cosine similarity, so downstream scoring stays a plain dot product) and a fixed η=0.10 explicit-swipe step (existing: `1/weight` incremental mean, which already decays the step size as a centroid matures). Confirmed with the user to **keep the existing math unchanged** rather than rewrite tested/documented behavior to match the letter of a spec that was, in effect, redescribing what already shipped. What was genuinely missing — a persisted calibration/training signal and `[AI-Stylist-ML]`-tagged diagnostic logging — was added on top:

- **`VisualClusterUpdater.update` now returns the nudge's drift percentage** (`‖c_new - c_old‖ * 100`, both sides unit-length) — `nil` when the call seeded a fresh centroid or hit the dimension-mismatch reseed path (no prior vector to diff against). Pure, no logging inside `Domain/VisualPreferenceProfile.swift` itself — callers log it.
- **`VisualPreferenceState` (`Models/SwipeDiscovery.swift`) gained `totalSwipes: Int`**, incremented only by `recordSwipe` (explicit deck swipes) — not by `applyImplicitSwipe`'s rating-derived nudges, which stay an ambient, gentler signal per the entry above, not a calibration action. Two computed properties: `calibrationProgress: Double` (linear `totalSwipes / 20`, capped at 1.0) and `isTrained: Bool` (`calibrationProgress >= 1.0`). Deliberately **not** gated on a `consecutiveStableUpdates`/"stability" dimension — the spec named such a field but never defined what "stable" meant, and the user's follow-up direction dropped it in favor of this simpler, purely swipe-count-driven meter.
- **`[AI-Stylist-ML]`-tagged logging**, added at three points per the request: (1) `Data/WardrobeRepository.swift`'s `recordSwipe`/`applyImplicitSwipe` log the drift percentage on every swipe (tagged `type=explicit`/`type=implicit`); (2) `recordSwipe` logs once when `isTrained` flips false→true; (3) `Domain/WardrobeCatalogBuilder.swift`'s `rank(_:history:)` logs each item's computed visual affinity bonus while ranking a slot, gated on the profile actually having centroids (a cold-start profile scoring everything 0 would just be log noise). New `Domain/MLLog.swift` holds the shared `Logger` — kept in `Domain/` rather than reusing `Services/PerfLog.swift` so `WardrobeCatalogBuilder` (Domain) doesn't need a `Services/` import (Domain/CLAUDE.md); `WardrobeRepository` (Data) reuses the same logger instance since it already depends on `Domain/` types here.
- **Calibration is surfaced in the UI as a gamified ring, not a raw log readout.** `Features/SwipeDiscovery/CalibrationRingView.swift` (new) is an Activity-ring-style meter — `CalibrationRingView` (the ring itself, spring-animated fill, checkmark at 100%) and `CalibrationProgressBadge` (ring + "Stylist Calibration" label, wrapped in `.premiumCard()`) — mounted at the top of `SwipeDiscoveryView`. `SwipeDiscoveryViewModel` exposes `calibrationProgress`/`isTrained`, refreshed via a best-effort `fetchVisualPreferenceState()` read on deck load and after every persisted swipe.
- **Vector math stays pure Swift, not Accelerate/vDSP.** Confirmed with the user — the existing `l2Normalized`/`cosineSimilarity` loops are already NaN-safe and tested, and 768-dimension vectors are cheap enough that a vDSP rewrite wasn't worth the diff against tested behavior for this pass.

## Decisions ("What Would You Change?" Checklist — Level 3, 2026-07-15)

Closes the second of three items tracked in this file's Deferred section: the Phase 1 addendum shipped Level 1/2 dimension-based outfit feedback but stopped short of "the 'What would you change?' checklist (Level 3)." No prior spec existed anywhere in the repo/docs for what the checklist's actual options should be — the seven below were defined fresh, each mapped 1:1 onto an existing (or newly-fed) affinity dimension, following this file's own stated design philosophy ("every dimension wired to a specific affinity, not captured-and-deferred").

- **Seven checklist options, six dimensions.** `Models/FeedbackEvent.swift`'s new `OutfitChangeReason` enum: Too Formal / Too Casual → `formalityAffinity` (mutually exclusive — selecting one clears the other, `RateCombinationViewModel.toggleChangeReason`), Wrong Colors → `colorVibeAffinity`, Wrong Pattern → `patternAffinity`, Not My Style → `styleTagAffinity`, Didn't Fit Right → `silhouetteAffinity`, Wrong For The Weather → `fabricWeightAffinity`.
- **"Wrong Pattern" is the first outfit-level entry point into `patternAffinity`.** Every other dimension already had an outfit-level Level 2 star question feeding it; pattern didn't (only item-level `ItemRating.patternLike` fed that map). `Domain/AttributePreferenceProfile.swift`'s `OutfitDimensionRatedAttributes` gained `pattern: GarmentPattern` (the item's actual pattern) and `patternDissatisfaction: Double?` (`nil` unless flagged), folded into `patternSums` in `build(from:outfitDimensionRatings:...)` identically in shape to the other five dimensions.
- **A flagged reason clamps, it doesn't replace.** `Data/WardrobeRepository.swift.fetchFeedbackHistory()`'s per-item `OutfitDimensionRatedAttributes` construction applies `min(existingNormalizedValue, 0.1)` for every flagged reason's dimension — so a user who gives a high Color Harmony star but also flags "Wrong colors" (e.g. rating charitably overall while still pointing at one specific miss) still contributes a strongly negative `colorVibeAffinity` signal, rather than the checklist being purely redundant with a low star.
- **No new scoring primitive.** The clamp reuses the exact same accumulation/shrinkage pipeline every other dimension already goes through (`AttributePreferenceProfile.build`) — this was a deliberate constraint per this file's design philosophy, not an oversight.
- **UI**: `Features/Rating/RateCombinationView.swift` gained a "What Would You Change?" `Form` section (`ChangeReasonChecklist`, a plain multi-select list of toggleable rows) between the Level 2 questions and the Favorite/Weakest picker.
- **`SchemaV7`** (`Models/SchemaMigrations.swift`) — `OutfitFeedback.changeReasonsRaw: [String] = []`, purely additive, `.lightweight` migration; `SchemaV6.OutfitFeedback` gained the frozen pre-`changeReasonsRaw` nested snapshot this requires.

## Decisions (Multi-Accessory Outfits, 2026-07-15)

Closes the last of three items tracked in this file's Deferred section — "multi-accessory support (letting an outfit hold several simultaneous accessories)" was explicitly scoped out of the 2026-07-14 prompt-restructure pass as needing "a schema/engine/validator/SwiftData migration across ~12 files."

- **Investigation found the gap narrower than "one accessory total."** Cross-slot accessorizing already worked — `Slot` already has three independent accent cases (`headwear`, `accessory`, `bag`), each 0-1. The actual constraint was that `.accessory` is an explicit catch-all bucket ("belt, scarf, tie, watch, or sunglasses... one per outfit, not several simultaneously," per `StylistBrain.swift`/`OutfitRecommendationService.swift`'s prior prose) — so a request like "bag + belt + jewelry" collided only on the belt/jewelry pair sharing one slot; the bag was always fine.
- **Fix is additive, not a redesign.** The primary `.accessory` slot (and every other slot's `[Slot: WardrobeItem]` dictionary shape) is untouched — zero risk to existing behavior or test coverage. A bounded **supplementary accessories** list (`FashionKnowledgeConstants.DressCode.maxSupplementaryAccessories = 2`) was added as a wholly separate field at every layer: wire (`RecommendedOutfitWire.supplementaryAccessoryIDs`, decoded through the existing dynamic-key `Codable` container under a fixed `supplementary_accessory_ids` key), JSON schema (a capped array property — an empty array natively expresses "nothing extra," no required-but-nullable hack needed the way scalar optional slots require), validator (`OutfitCombination.supplementaryAccessories`), persistence (`SavedCombination.supplementaryAccessoryItemIDs`/`Labels`, new `SchemaV8`), prompt (`StylistBrain`), and UI (`OutfitCardView`).
- **The validator's existing per-slot `Slot.allCases` loop is untouched** — supplementary accessories are resolved as one explicit extra step alongside it, not woven into it, per `Domain/CLAUDE.md`'s guardrail against ad hoc per-slot special-casing inside that generic loop. The duplicate-id check was extended to cover supplementary ids too (can't repeat the primary accessory or each other).
- **Over-cap supplementary ids are truncated, not rejected.** The schema's `maxItems` should prevent this from the model side, but the validator defensively takes only the first `maxSupplementaryAccessories` rather than hard-failing the whole outfit over a minor schema violation.
- **Scenario-gated in the prompt, same tier as the existing single-accessory rule.** Business/interview/formal-event scenarios must leave `supplementary_accessory_ids` empty (same ceiling as the existing "at most one subtle accessory" rule); only casual/going-out scenarios may use up to the cap.
- **`Features/JobQueue/JobQueueStore.saveCombination`** and **`Features/Combinations/CombinationsViewModel.resolveItems(for:)`** are the two integration points that build/resolve a `SavedCombination` from/to real items — both updated to carry the new fields; Manual Pairing saves (top+bottom only) are unaffected since they never populate `supplementaryAccessories`.

## Decisions (Impression/Selection Event Capture, 2026-07-15)

Closes one of three deferred items below (multi-accessory outfits and the Level 3 feedback checklist are tracked separately as they land). The app already logged ML drift (`[AI-Stylist-ML]`) but never recorded which of several shown candidate outfits the user actually acted on vs. ignored.

- **New event-sourced `Models/RecommendationImpressionEvent.swift`**, mirroring `SwipeEvent`'s shape rather than `FeedbackEvent.swift`'s tables — there's no durable `SavedCombination.id` to key against at impression time, since an ignored candidate is never persisted. One row is inserted per candidate the moment `DailyAssistantViewModel.sendTurn` resolves a successful round, keyed by the in-memory `OutfitCombination.id` (stable for the lifetime of that value within the round) with `rank` = its position in the LLM's already-strongest-to-weakest sort.
- **`startTryOn` is the "selection" gesture.** Among the concrete "pick" actions in the app, tapping "How does it look on me?" is the one that corresponds to choosing one candidate over its shown siblings — `DailyAssistantViewModel.startTryOn` marks that impression row's `selectedAt`. The later save/like flow (`JobQueueStore.saveCombination`) is decoupled from round context entirely and untouched by this change.
- **Both new repository methods (`recordImpressions`/`recordSelection`) are best-effort**, called via `try?` exactly like `applyImplicitSwipe` — a logging/audit trail, never a gate on the conversation flow — and log under the existing `[AI-Stylist-ML]` tag (`Domain/MLLog.swift`).
- **Not wired into any ranker this phase** — same posture as `OutfitRecommendationValidator.RejectionReason`: the data now exists so a future "which of our recommendations actually land" investigation has something to query, but `AttributePreferenceProfile`/`PairCompatibilityScoring` don't read it yet.

## Deferred (later phases)

Full knowledge-module extraction (separate Color Theory / Fabric Science / Dress Code / Silhouette / Footwear modules with individual applicability predicates and prompt-budget eviction), the validator's contextual (Tier B) and repair (Tier C) tiers, a named versioned Recommendation Rubric with per-dimension subscores, exemplar-based taste injection, and controlled vocabularies for the rich garment attributes are all real next steps but were not done in this pass — each touches either the live structured-output contract with `google/gemini-3.1-flash-lite` or the well-tested numeric scoring engine, and is better done incrementally with its own regression coverage rather than in one sweep. See the full architecture design for the target end-state; this file tracks only what's actually shipped.
