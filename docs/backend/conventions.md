# Backend Conventions

`backend/functions` (Firebase Cloud Functions, Node/TypeScript) exists — see `docs/backend/architecture.md` for the design and `backend/README.md` for setup/deploy. These conventions carry over what was already established on the iOS side rather than inventing new ones:

- **Typed schemas at every border** — the iOS client does this with `Codable` + explicit `CodingKeys` (`docs/approach/conventions.md`); the backend does the equivalent with Zod, but only a **loose top-level shape check** (e.g. `model`/`messages` present and typed) in `backend/functions/src/routes/*.ts` — not a deep mirror of each iOS service's JSON Schema. Deep validation would duplicate business logic the client already owns (see "No business logic in the proxy" below).
- **No business logic in the passthrough proxy routes.** The 5 passthrough routes' job is exactly "verify the caller, hold the API key, forward the request" — the deterministic scoring engine stays on-device (`docs/backend/architecture.md` § What NOT to do), and so does all prompt/schema construction (`Config/ModelConfig.swift`). A passthrough route that starts reimplementing scoring, retrieval, or prompt-building logic has drifted from the intended architecture. Four routes (`/account/delete`, `/iap/verify`, `/entitlement/limits`, `/analytics/config`) are **deliberate exceptions** that contain server-side business logic — each is individually justified in its route file's doc comment and in `docs/backend/architecture.md`. Before adding a new exception, verify the route genuinely requires Admin SDK privileges or server-only-write Firestore access that the client can never safely hold.
- **Testing:** Vitest (`backend/functions/test/`), not Jest — mock `firebase-admin` and `fetch`, never make live network calls in a test. Cover middleware rejection paths (missing/invalid Auth token, rate-limit/quota boundary) and each route's forward/passthrough behavior (`backend/functions/test/middleware.test.ts`, `governance.test.ts`, `routes.test.ts` are the reference examples).
- **Auth verification lives in middleware** (`backend/functions/src/middleware/`), applied once in `backend/functions/src/app.ts`, not repeated per route. App Check is deferred (paid Apple Developer account required for App Attest) — see `docs/decisions/resolved-v1.md`.
- **Secrets** via Firebase Functions secret manager (`firebase functions:secrets:set`, declared with `defineSecret` in `backend/functions/src/secrets.ts`) — never an `.env` file, never committed.
- **Logging:** route, uid, upstream status, and latency only — never full request bodies (they carry prompt text and base64 image data).

## AI model hotfix via Firebase Remote Config

Every OpenRouter model constant in `Config/ModelConfig.swift` (`textToText`, `imageToText`, `imageToImage`, `imageEdit`) reads from Firebase Remote Config (`Vision_clother/Config/RemoteConfigManager.swift`), not a hardcoded Swift literal — plus `textToText`'s own payload knobs (fallback model, temperature, max tokens, strict-JSON-schema). This is a client-only change — `openrouterChat.ts`'s `bodySchema` is already `z.object({ model, messages }).passthrough()`, so `temperature`/`max_tokens`/`response_format` pass through untouched with no backend edit required (per "No business logic in the passthrough proxy routes" above).

**Why:** an OpenRouter model can be deprecated or start misbehaving without warning; before this, recovering required an app-binary change and a full App Store release. Now the affected value can be changed in the Firebase Console and takes effect for any client that fetches Remote Config next (each launch, subject to the 1-hour `minimumFetchInterval`) — no build, no review, no release.

**Procedure:** Firebase Console → Remote Config → edit the relevant parameter(s) below → Publish. Every parameter has a client-side default (`RemoteConfigManager.Defaults`), so a client that never fetches (offline, first launch, fetch failure) still runs correctly — this is a hotfix path, not a required dependency for the app to function. **Live in the `visionclother` project** as of 2026-07-21 (published via the Firebase MCP `remoteconfig_update_template` tool) with values identical to the defaults below, so publishing today is a no-op — the parameters just become visible/editable in the Console for the next actual hotfix.

| Parameter | Type | Default | Consumed by |
|---|---|---|---|
| `ai_primary_model_name` | String | `google/gemini-3.1-flash-lite` | `ModelConfig.textToText` — the model every text-in/JSON-out call (recommendation, intent extraction, Q&A) sends first. |
| `ai_fallback_model_name` | String | `openai/gpt-5-mini` | `ModelConfig.textToTextFallback` — `OpenRouterOutfitRecommendationService`'s retry model when the primary model's structured-output attempt comes back empty/malformed/rejected. |
| `ai_temperature` | Number | `0` | `ModelConfig.temperature` — request `temperature` for the recommendation call. |
| `ai_enable_strict_json_schema` | Boolean | `true` | `ModelConfig.enableStrictJSONSchema` — whether the recommendation call's `response_format: json_schema` requests `strict: true`; a per-model escape hatch for one that accepts the schema mode but rejects strict enforcement. |
| `ai_max_tokens` | Number | `4096` | `ModelConfig.maxTokens` — request `max_tokens` for the recommendation call. |
| `ai_image_to_text_model_name` | String | `minimax/minimax-m3` | `ModelConfig.imageToText` — garment tagging (`OpenRouterVisionMetadataExtractionService`) and onboarding-portrait style-profile derivation (`OpenRouterUserProfileDerivationService`); one image in, JSON text out. |
| `ai_image_to_image_model_name` | String | `google/gemini-3.1-flash-lite-image` | `ModelConfig.imageToImage` — virtual try-on render (`OpenRouterTryOnRenderService`); base portrait + garment images in, a generated photo out. Currently Google's Gemini image model (community nickname "nano banana"). |
| `ai_image_edit_model_name` | String | `google/gemini-3.1-flash-lite-image` | `ModelConfig.imageEdit` — AI-assisted background removal (`OpenRouterBackgroundIsolationService`); single reference image + edit prompt, not a multi-image compose. Independently tunable from `ai_image_to_image_model_name` even though it currently defaults to the same model. |

Import this table into the Firebase Console by hand (Remote Config has no bulk-JSON-import UI for a fresh project) — parameter key, type, and default value map directly to the columns above. Keep parameter `description` fields plain ASCII (no em dashes, no embedded smart quotes) — the Remote Config REST API silently rejected a couple of publish attempts during setup until they were simplified.
