# Backend Conventions

`backend/functions` (Firebase Cloud Functions, Node/TypeScript) exists тАФ see `docs/backend/architecture.md` for the design and `backend/README.md` for setup/deploy. These conventions carry over what was already established on the iOS side rather than inventing new ones:

- **Typed schemas at every border** тАФ the iOS client does this with `Codable` + explicit `CodingKeys` (`docs/approach/conventions.md`); the backend does the equivalent with Zod, but only a **loose top-level shape check** (e.g. `model`/`messages` present and typed) in `backend/functions/src/routes/*.ts` тАФ not a deep mirror of each iOS service's JSON Schema. Deep validation would duplicate business logic the client already owns (see "No business logic in the proxy" below).
- **No business logic in the passthrough proxy routes.** The 5 passthrough routes' job is exactly "verify the caller, hold the API key, forward the request" тАФ the deterministic scoring engine stays on-device (`docs/backend/architecture.md` ┬з What NOT to do), and so does all prompt/schema construction (`Config/ModelConfig.swift`). A passthrough route that starts reimplementing scoring, retrieval, or prompt-building logic has drifted from the intended architecture. Four routes (`/account/delete`, `/iap/verify`, `/entitlement/limits`, `/analytics/config`) are **deliberate exceptions** that contain server-side business logic тАФ each is individually justified in its route file's doc comment and in `docs/backend/architecture.md`. Before adding a new exception, verify the route genuinely requires Admin SDK privileges or server-only-write Firestore access that the client can never safely hold.
- **Testing:** Vitest (`backend/functions/test/`), not Jest тАФ mock `firebase-admin` and `fetch`, never make live network calls in a test. Cover middleware rejection paths (missing/invalid Auth token, rate-limit/credit boundary) and each route's forward/passthrough behavior (`backend/functions/test/middleware.test.ts`, `governance.test.ts`, `creditGate.test.ts`, `pricing.config.test.ts`, `routes.test.ts` are the reference examples).
- **Auth verification lives in middleware** (`backend/functions/src/middleware/`), applied once in `backend/functions/src/app.ts`, not repeated per route. App Check is deferred (paid Apple Developer account required for App Attest) тАФ see `docs/decisions/resolved-v1.md`.
- **Secrets** via Firebase Functions secret manager (`firebase functions:secrets:set`, declared with `defineSecret` in `backend/functions/src/secrets.ts`) тАФ never an `.env` file, never committed.
- **Logging:** route, uid, upstream status, and latency only тАФ never full request bodies (they carry prompt text and base64 image data). Storage-triggered functions (no inbound HTTP request to key off) log the Storage object's `objectName`/`generation` in place of `route`/`requestId` тАФ same rule against logging raw bytes applies (`wardrobeImageProcessing.ts` never logs image bytes/base64, only sizes/dimensions/durations).

## AI model hotfix via Firebase Remote Config

Every OpenRouter model constant in `Config/ModelConfig.swift` (`textToText`, `imageToText`, `imageToImage`, `imageEdit`) reads from Firebase Remote Config (`Vision_clother/Config/RemoteConfigManager.swift`), not a hardcoded Swift literal тАФ plus `textToText`'s own payload knobs (fallback model, temperature, max tokens, strict-JSON-schema). This is a client-only change тАФ `openrouterChat.ts`'s `bodySchema` is already `z.object({ model, messages }).passthrough()`, so `temperature`/`max_tokens`/`response_format` pass through untouched with no backend edit required (per "No business logic in the passthrough proxy routes" above).

**Why:** an OpenRouter model can be deprecated or start misbehaving without warning; before this, recovering required an app-binary change and a full App Store release. Now the affected value can be changed in the Firebase Console and takes effect for any client that fetches Remote Config next (each launch, subject to the 1-hour `minimumFetchInterval`) тАФ no build, no review, no release.

**Procedure:** Firebase Console тЖТ Remote Config тЖТ edit the relevant parameter(s) below тЖТ Publish. Every parameter has a client-side default (`RemoteConfigManager.Defaults`), so a client that never fetches (offline, first launch, fetch failure) still runs correctly тАФ this is a hotfix path, not a required dependency for the app to function. **Live in the `visionclother` project** as of 2026-07-21 (published via the Firebase MCP `remoteconfig_update_template` tool) with values identical to the defaults below, so publishing today is a no-op тАФ the parameters just become visible/editable in the Console for the next actual hotfix.

| Parameter | Type | Default | Consumed by |
|---|---|---|---|
| `ai_primary_model_name` | String | `google/gemini-3.1-flash-lite` | `ModelConfig.textToText` тАФ the model every text-in/JSON-out call (recommendation, intent extraction, Q&A) sends first. |
| `ai_fallback_model_name` | String | `openai/gpt-5-mini` | `ModelConfig.textToTextFallback` тАФ `OpenRouterOutfitRecommendationService`'s retry model when the primary model's structured-output attempt comes back empty/malformed/rejected. |
| `ai_temperature` | Number | `0` | `ModelConfig.temperature` тАФ request `temperature` for the recommendation call. |
| `ai_enable_strict_json_schema` | Boolean | `true` | `ModelConfig.enableStrictJSONSchema` тАФ whether the recommendation call's `response_format: json_schema` requests `strict: true`; a per-model escape hatch for one that accepts the schema mode but rejects strict enforcement. |
| `ai_max_tokens` | Number | `4096` | `ModelConfig.maxTokens` тАФ request `max_tokens` for the recommendation call. |
| `ai_image_to_text_model_name` | String | `minimax/minimax-m3` | `ModelConfig.imageToText` тАФ garment tagging (`OpenRouterVisionMetadataExtractionService`) and onboarding-portrait style-profile derivation (`OpenRouterUserProfileDerivationService`); one image in, JSON text out. |
| `ai_image_to_image_model_name` | String | `google/gemini-3.1-flash-lite-image` | `ModelConfig.imageToImage` тАФ virtual try-on render (`OpenRouterTryOnRenderService`); base portrait + garment images in, a generated photo out. Currently Google's Gemini image model (community nickname "nano banana"). |
| `ai_image_edit_model_name` | String | `google/gemini-3.1-flash-lite-image` | `ModelConfig.imageEdit` тАФ AI-assisted background removal (`OpenRouterBackgroundIsolationService`); single reference image + edit prompt, not a multi-image compose. Independently tunable from `ai_image_to_image_model_name` even though it currently defaults to the same model. |

Import this table into the Firebase Console by hand (Remote Config has no bulk-JSON-import UI for a fresh project) тАФ parameter key, type, and default value map directly to the columns above. Keep parameter `description` fields plain ASCII (no em dashes, no embedded smart quotes) тАФ the Remote Config REST API silently rejected a couple of publish attempts during setup until they were simplified.

## Server-side model allowlist

Because the client's `model` values above are hotfixable with no rebuild, `openrouterChat.ts` and `openrouterImages.ts` cannot rely on the client to only ever request a small, safe set of models тАФ a compromised or scripted client could send any string, and combined with free/unlimited Firebase anonymous guest accounts (`AuthService.ensureGuestSession()`), that's a billing-abuse surface (the project's own OpenRouter key pays for whatever model is requested). `backend/functions/src/modelAllowlist.ts` enforces an **exact-match** allowlist (`assertModelAllowed`) тАФ not a provider-prefix match тАФ since the goal is capping worst-case spend per request, and prefix-matching would still let an allowed provider's most expensive model through.

The allowlist lives in Firestore (`config/openrouterModels`, `{ allowedModels: string[] }`), not hardcoded, specifically so it can track the client's own Remote-Config hotfixes without a backend redeploy: when a new model is adopted in `ModelConfig.swift`, add it to `config/openrouterModels` (Firebase Console or an Admin SDK/MCP tool, takes effect within the 5-minute cache TTL, no redeploy) тАФ or, failing that, to `DEFAULT_ALLOWED_MODELS` in `modelAllowlist.ts` (requires a redeploy). Do at least one, or the hotfixed model starts getting `403 model_not_allowed` in production. Current allowlist mirrors the table above plus the documented alternates in `ModelConfig.swift`'s comments: `google/gemini-3.1-flash-lite`, `openai/gpt-5-mini`, `minimax/minimax-m3`, `google/gemini-3.1-flash-lite-image`, `qwen/qwen3-vl-30b-a3b-instruct`, `bytedance-seed/seedream-4.5`, `google/gemini-2.5-flash-image`.

This is the first global (non-`users/{uid}`) Firestore config doc in this codebase тАФ `backend/firestore.rules` denies all client read/write on it, same as `processedTransactions`/`idempotencyKeys`. Any future need for backend-owned, ops-editable config should follow this same pattern (top-level collection, Admin-SDK-only, hardcoded fallback for zero-seeding correctness, warm-instance in-memory cache) rather than inventing a new one.

## Credit & tier management engine config

`backend/functions/src/pricing.config.ts` is the second adopter of the pattern above: `config/pricing` (`{ operationCosts, tierConfigs }`), Admin-SDK-only, hardcoded `DEFAULT_OPERATION_COSTS`/`DEFAULT_TIER_CONFIGS` fallback, 1-minute warm-instance TTL cache (deliberately shorter than `modelAllowlist.ts`'s 5 minutes тАФ price/cap changes need to propagate faster), stale-cache-preferred-on-failure. It replaced the earlier `entitlementLimits.ts` (plain hardcoded TS constants, no Firestore backing, no runtime-editable path) тАФ ops can now retune an operation's credit cost or add a brand-new tier (e.g. a temporary `ULTRA_PRO`) by writing to `config/pricing`, with zero backend redeploy and zero change to `middleware/creditGate.ts`'s enforcement logic. `backend/firestore.rules`' `itemCap()` also reads this doc live via `get()` (Rules' `get()`/`exists()` are not subject to the target doc's own `allow read`/`write` rules, so this is safe despite `config/pricing` itself being Admin-SDK-only for direct client access) тАФ a tier added only via Firestore is fully enforced at both the credit-gate and rules layer with zero redeploy. See `docs/backend/architecture.md`'s "Credit gate middleware" section for the full design.
