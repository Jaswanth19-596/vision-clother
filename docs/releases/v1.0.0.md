# Vision Clother — v1.0.0

Snapshot of what shipped in V1, what's explicitly out of scope, and what a V2 pass should prioritize. Read this alongside `docs/timeline.md` (chronological change log) and `docs/decisions/*` (the *why* behind each call) — this doc is the *what's actually done* summary those two don't give you in one place.

## What V1 Achieved

**Core recommendation pipeline (PRD §2.1a, §3.7).** The LLM (`google/gemini-3.1-flash-lite` via OpenRouter, `Config/ModelConfig.swift`) is the primary outfit recommender: free-text prompt + a bounded, text-only wardrobe catalog (`Domain/WardrobeCatalogBuilder.swift`, capped at 150 items, slot-balanced) + derived User Style Profile + weather + color-theory guidance in, up to 5 ranked outfit picks out. `Domain/OutfitRecommendationValidator.swift` is the deterministic trust boundary — every pick is checked against real/correctly-slotted/non-duplicate/non-Ghost items before it reaches the user, then re-scored by the original deterministic engine (`Domain/OutfitRecommendationEngine.swift`, `Domain/PairCompatibilityScoring.swift`), which remains a permanent, fully-offline fallback if the LLM call fails or validation yields nothing.

**Stylist Intelligence Engine (`docs/decisions/stylist-intelligence-engine.md`).** A lexicographic 6-tier Decision Hierarchy (hard constraints → dress code → weather → preferences → visual cohesion → aesthetic trend) drives both the prompt and the deterministic scorer off shared constants (`Domain/FashionKnowledgeConstants.swift`). Resolved constraints are self-reported by the same recommendation call (no second LLM round-trip); learned taste is injected symmetrically (likes *and* dislikes); rejection reasons are captured (`RejectionReason`, not yet wired to telemetry).

**Wardrobe management.** Seven garment slots (`top`/`bottom`/`footwear`/`outerwear`/`headwear`/`accessory`/`bag`), each with `isRequired`/`hasGhostDefault` behavior rather than ad hoc special-casing. Ingestion runs as a background job (`Features/JobQueue/JobQueueStore.swift`): Gemini foreground isolation → on-device Vision cutout → vision-LLM tagging → save, with graceful per-stage fallback to the previous stage's output on failure. Ghost Elements (`Domain/GhostElementProvider.swift`) backfill empty required slots so a new user with an empty closet still gets complete outfit suggestions on day one.

**Virtual try-on.** `OpenRouterTryOnRenderService` composites the user's portrait with every populated slot's garment photo into one generated image (Gemini image models via OpenRouter chat completions). `Services/CachedTryOnRenderService.swift` reuses an existing saved render for the same portrait+item-set instead of re-generating. Runs as an independent background job per request — multiple renders can be in flight concurrently, each cancellable individually.

**Manual Outfit Pairing** (`Features/Pairing/`) — direct top/bottom selection with the same try-on renderer, independent of the recommendation engine.

**Conversational refinement loop** (Stylist Intelligence Engine ADR Phase 2) — up to 2 clarifying follow-up turns before the model is forced to decide; free-text refinement ("no bag, something else") is uncapped and reuses the same conversation.

**Prospective Purchase Evaluation** ("is this a good buy?") — tag a photo of an item not yet owned, force every candidate outfit around it (`mustIncludeItemID`), and let the user Add to Closet or discard, without ever persisting anything until they choose to.

**Three-tier feedback + learned taste** (PRD §3.6). `OutfitFeedback` (Level 1 quick + Level 2 dimension-based ratings + Level 3 "What would you change?" checklist), `ItemFeedback`, `PairFeedback`, and per-item `ItemRating` (Fit/Comfort/Color/Pattern/Formality Fit/Style Identity/Wear Again) all feed `Domain/AttributePreferenceProfile.swift`'s Bayesian-shrunk affinity maps (color, pattern, formality, style tag, silhouette, fabric weight), surfaced in the Profile tab as "Your Taste, In Words" (`Domain/TasteSynthesis.swift`) rather than raw numbers.

**Swipe-to-Learn Visual Taste.** A Tinder-style stock-photo swipe deck (`Features/SwipeDiscovery/`, Pexels-backed) builds on-device Vision feature-print embeddings and an online k-means (up to 3 centroids per liked/disliked side) with no bundled ML model and no network call for the embedding itself. Feeds both a re-ranking term on the deterministic engine and embedding-ranked catalog truncation. Item ratings also close the loop as gentler "implicit swipes." A gamified calibration ring shows progress toward a trained profile.

**Saved Combinations tab** — durable image + item-ID persistence for any liked try-on result, mirrored to the Photos library, independent of whether the source items are later deleted from the closet (denormalized labels).

**Multi-accessory outfits** — a bounded (max 2) supplementary-accessory list alongside the primary single-accessory slot, so "bag + belt + jewelry" can be recommended together, gated by scenario formality in the prompt.

**Persistence.** 8 SwiftData schema versions shipped (`Models/SchemaMigrations.swift`) — one `.custom` migration (V1→V2, the original slot-expansion) and seven `.lightweight` additive migrations, all landed incrementally with dedicated migration tests before dependent features were built on top.

**Design system** — a shared `DesignSystem/` module (spacing/radius/shadow tokens, button styles, haptics on critical actions) rolled out across every feature folder, replacing ad hoc per-file styling.

## Known Limitations

- **No backend exists.** OpenRouter/Fal/Pexels API keys ship inside the app bundle (`Config/Secrets.plist`) and every AI call goes device-to-provider directly. Explicitly documented as dev-only (`docs/decisions/resolved-v1.md`, `docs/backend/architecture.md`) — **not safe to distribute (TestFlight/App Store) as-is.**
- **Local-only persistence, no account system.** SwiftData is the only store; there is no multi-device sync, no server-side backup beyond whatever the OS/iCloud device backup provides, and no way to recover from a corrupted store other than a full local reset (`Vision_clotherApp.swift`'s `recreatingContainerAfterStoreReset` — deletes and recreates the store on any `ModelContainer` init failure, with no user-facing warning).
- **`fetchFeedbackHistory()` cost still scales with wardrobe/history size.** The 2026-07-14 "Bottleneck C" pass added time-window predicates and moved the affinity-profile computation to `Task.detached`, but the per-item Vision embedding computation for cache-misses still runs in a serial loop on the `@MainActor`-isolated repository — a large uncached wardrobe still means a slow first recommendation call.
- **WeatherKit is not wired.** `WeatherProvider.swift`'s `WeatherKitWeatherProvider` always returns `nil`; `ServiceFactory` defaults to the mock provider everywhere.
- **No dedicated onboarding flow.** The base portrait is captured opportunistically (first Manual Pairing or Profile-tab visit), not a first-run flow.
- **Architecture docs describe a provider no longer in use.** `docs/decisions/resolved-v1.md`, `docs/ios/architecture.md`, and `docs/backend/architecture.md` all describe Fal + `fashn/tryon/v1.6` for try-on; the actual implementation is `OpenRouterTryOnRenderService` against Gemini image models. No `Fal*.swift` file exists anywhere in the codebase.
- **No concurrency cap on the background job queue.** Uploads and try-on jobs each spawn an independent, unbounded `Task` — no client-side rate limiting against the third-party APIs.
- **No telemetry or crash reporting.** `RejectionReason` (validator) and `RecommendationImpressionEvent`/selection data are captured and logged locally (`[AI-Stylist-ML]` tag) but not wired to any analytics sink — there's no way to observe real-world recommendation quality, latency, or failure rates in aggregate.
- **Single garment per photo.** Multi-item photo splitting was explicitly scoped out; each upload tags exactly one garment.
- **No bulk re-classification tool.** Mis-tagged items (e.g. from the pre-2026-07-14 slot-drift bug) must be corrected one at a time via `ItemDetailView` → Edit.
- **Accent-item cap.** An outfit can carry at most one primary accessory + up to 2 supplementary accessories (`FashionKnowledgeConstants.DressCode.maxSupplementaryAccessories`), plus one headwear and one bag — not an arbitrary accessory count.

## Deferred Features

Tracked explicitly in `docs/decisions/stylist-intelligence-engine.md`'s Deferred section and scattered scope notes across the timeline — real next steps, not forgotten:

- Full knowledge-module extraction (separate Color Theory / Fabric Science / Dress Code / Silhouette / Footwear modules with individual applicability predicates and prompt-budget eviction).
- The validator's contextual (Tier B) and repair (Tier C) tiers — today the validator only does hard reject/accept, no "fix and retry."
- A named, versioned Recommendation Rubric with per-dimension subscores (today's `rationale.confidence` is a single 0-100 scalar).
- Exemplar-based taste injection (today's taste injection is threshold-based prose, not concrete outfit examples).
- Controlled vocabularies for the rich garment attributes (`garment_subtype`, `fit`, `silhouette`, `material`, `texture` are currently free-text from the vision-tagging LLM, not a fixed enum).
- Real WeatherKit integration (currently mocked).
- Dedicated onboarding flow for base-portrait capture.
- A thin proxy backend (architecture already sketched in `docs/backend/architecture.md`, not yet built).
- Wiring the already-captured `RejectionReason`/impression-selection data into any actual ranking or debugging tool.

## V2 Goals

Proposed priorities, in rough order — none of this is committed, but this is where the "known limitations" above point most directly:

1. **Ship the proxy backend.** This is the single blocking item for any real distribution — hold OpenRouter/Fal/Pexels keys server-side, mirror the existing client-side service protocols (`IntentExtractionService`, `OutfitRecommendationService`, `TryOnRenderService`, `VisionMetadataExtractionService`), and add basic rate limiting/cost controls per user. Everything else on this list is secondary to this.
2. **Reconcile the docs with reality.** Update `resolved-v1.md`/`docs/ios/architecture.md`/`docs/backend/architecture.md` to reflect OpenRouter/Gemini as the current try-on provider instead of the superseded Fal decision.
3. **Close the telemetry loop.** Ship the already-instrumented `RejectionReason` and impression/selection data somewhere queryable, so recommendation quality and failure modes are observable instead of only locally logged.
4. **Fix the recommendation-latency floor.** Parallelize the weather/inventory/history/profile fetches in `DailyAssistantViewModel.resolveOutfits` (currently sequential `await`s with no dependency between them), and move the per-item embedding computation off the main actor with concurrent (not serial) execution.
5. **Add a data-loss safeguard.** Before another schema version ships, add at minimum a log line (ideally a lightweight local export attempt) when `recreatingContainerAfterStoreReset` triggers, so a migration bug doesn't silently destroy a user's only copy of their wardrobe/history with no record it happened.
6. **Real WeatherKit + a first-run onboarding flow.** Both are scoped, low-risk, and currently mocked/opportunistic respectively.
7. **Multi-device / account layer**, once the proxy backend exists to hang it off of — out of scope until then.
